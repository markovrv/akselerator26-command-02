# Publishing Guide

How to prepare this project for publishing via the Publish tab.

## Overview

The platform supports four service types:
1. **Frontend** — static sites and SPAs
2. **Backend** — server apps in Docker containers (any language)
3. **Database** — managed instances (PostgreSQL, MySQL, MongoDB, Redis, ClickHouse, Kafka, RabbitMQ, OpenSearch)
4. **S3 Storage** — S3-compatible object storage buckets (hot or cold)

No platform-specific config files are needed. All deployment configuration is handled via the `promto.yaml` file (see below).

---

## Frontend

### Static frontend (SPA)
Static sites built and served as static files behind Nginx with auto SSL.

**Supported frameworks:** `react`, `preact`, `vue`, `angular`, `svelte`, `nuxt`, `next.js`, `ember`, `another`

**What to provide:**
- `package.json` with dependencies and a `"build"` script
- Source code

No Dockerfile needed. The platform runs the build command and serves the output directory.

**Available preset (ru-1):**

| Preset ID | Disk |
|-----------|------|
| 2785      | 2 GB |

Frontend apps are static — no CPU/RAM allocation needed. Files are served via Nginx with auto SSL.

**Key settings in promto.yaml:**
- `type`: `"frontend"`
- `framework`: auto-detected from package.json (e.g. `"react"`, `"vue"`)
- `build_command`: e.g. `"npm install && npm run build"`
- `publish_dir`: output directory where `index.html` lives after build
  - Vite projects: `"dist"`
  - Create React App: `"build"`
  - Nuxt static: `".output/public"`

### SSR frontend (server-rendered)
Frameworks like **Next.js (SSR/SSG), Nuxt (SSR), SvelteKit, Remix** require a running Node.js server — they can NOT be deployed as static files. These must be deployed as **backend** services with Docker.

**How to identify SSR:**
- Next.js with `getServerSideProps`, API routes, or `output: "standalone"` → SSR
- Nuxt without `target: "static"` → SSR
- SvelteKit with `adapter-node` → SSR
- Any frontend that needs a running server process → SSR

**For SSR frontends, use `type: "backend"` with `framework: "docker"`** and provide a Dockerfile. Example Next.js SSR:

```dockerfile
FROM node:20-slim
WORKDIR /app
COPY package*.json .
RUN npm ci
COPY . .
RUN npm run build
EXPOSE 3000
CMD ["npm", "start"]
```

promto.yaml for SSR:
```yaml
  web:
    friendly_name: "Веб-приложение (Next.js SSR)"
    type: backend        # NOT frontend — it needs a running server
    framework: docker
    build_command: "docker build -t app ."
    start_command: "npm start"
    root_dir: "web"
    env_file: ".env"
```

**Rule of thumb:** If the framework needs `npm start` or a running process after build → use `type: "backend"` with Docker. If it produces static HTML/JS/CSS files → use `type: "frontend"`.

## Backend

Server applications in Docker containers. **Any language or stack** works.

**Supported backend frameworks (Timeweb):** `docker`, `django`, `express`, `fastapi`, `flask`, `laravel`, `fastify`, `nest`, `gin`, `Spring`, `ASP.NET Core`, `phoenix`, `beego`, `hapi`, `celery`, `symfony`, `yii`

We always use `docker` as the framework — it's the universal option that works with any language/stack.

**IMPORTANT: A `Dockerfile` is REQUIRED for each Docker-based service (backends AND SSR frontends).**

### Dockerfile placement rules:
- **Single-service project:** Dockerfile goes in the project root (`/home/user/my-project/Dockerfile`)
- **Multi-service project:** **NEVER put Dockerfiles in the project root.** Each Docker-based service's Dockerfile MUST be inside its own subdirectory (e.g. `Go-back/Dockerfile`, `api/Dockerfile`, `web/Dockerfile`). Set `root_dir` in promto.yaml to match.

### Dockerfile rules:
1. **Must have `EXPOSE <port>`** — tells the platform which port the app listens on (defaults to 8080 if missing)
2. **App must bind to `0.0.0.0`** — not `localhost` or `127.0.0.1`
3. **Ports 80 and 443 are reserved** — use any other port (3000, 5000, 8000, 8080, etc.)

### Example Dockerfiles:

**Python (FastAPI):**
```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**Node.js (Express):**
```dockerfile
FROM node:20-slim
WORKDIR /app
COPY package*.json .
RUN npm ci --production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

**Go:**
```dockerfile
FROM golang:1.22 AS build
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o server .

FROM alpine:3.19
COPY --from=build /app/server /server
EXPOSE 8080
CMD ["/server"]
```

### Docker Compose support

The platform supports deploying via `docker-compose.yml` in the repository root. This allows running any application, including stacks not directly supported by the platform.

**docker-compose.yml rules:**
1. `Dockerfile` and `docker-compose.yml` must be in the root of the connected repository.
2. If a Dockerfile is in a subdirectory, specify the path explicitly:
```yaml
services:
  app:
    build:
      context: ./app                        # Path to source code folder
      dockerfile: ./docker/Dockerfile.prod  # Explicit path to Dockerfile
```
3. **Ports 80 and 443 are reserved** — do NOT use them as host ports. Use any other ports.
4. Only the **first service** in docker-compose.yml gets proxied to the main domain. Other services with host ports are accessible via `http://domain:port`.
```yaml
services:
  my-app:
    ports:
      - "9000:80"    # Proxied to main domain
  adminer:
    ports:
      - "8080:8080"  # Accessible at http://domain:8080
```
5. **Forbidden directives** (will cause deploy error): `cap_add`, `cgroup`, `cgroup_parent`, `devices`, `privileged`, `device_cgroup_rules`, `volumes`, `ipc`, `pid`, `security_opt`, `userns_mode`, `sysctls`
6. **Network mode `host` is forbidden.**

**Available presets (ru-1):**

| Preset ID | CPU | RAM  | Disk  |
|-----------|-----|------|-------|
| 1003      | 1   | 1 GB | 15 GB |
| 1005      | 1   | 2 GB | 30 GB |
| 1018      | 2   | 4 GB | 50 GB |
| 1020      | 4   | 8 GB | 80 GB |

**Key settings in promto.yaml:**
- `type`: `"backend"`
- `framework`: always `"docker"`
- `build_command`: e.g. `"docker build -t app ."`
- `start_command`: e.g. `"uvicorn main:app --host 0.0.0.0 --port 8000"`

## S3 Storage

S3-compatible object storage buckets (Timeweb S3). Fully compatible with the Amazon S3 API — works with any S3 client library (boto3, aws-sdk, etc.).

**Storage classes:**
- `hot` — fast access, for frequently used files (images, assets, uploads)
- `cold` — cheaper archival storage (pay-as-you-go), for backups and infrequently accessed data

**Available presets (ru-1):**

| Preset ID | Class | Disk     |
|-----------|-------|----------|
| 2667      | hot   | 1 GB     |
| 2669      | hot   | 10 GB    |
| 2671      | hot   | 100 GB   |
| 2673      | hot   | 250 GB   |
| 2623      | cold  | 1 TB     |

**Bucket types:**
- `private` — access only via API with access_key/secret_key
- `public` — publicly accessible via URL

**What you get after creation:**
- `hostname` — S3 endpoint (e.g. `s3.twcstorage.ru`)
- `access_key` — S3 access key ID
- `secret_key` — S3 secret access key
- `bucket_name` — the bucket name

**For the agent:** Use environment variables for S3 credentials, never hardcode them. Typical env vars: `S3_ENDPOINT`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_BUCKET`.

**Key settings in promto.yaml:**
- `type`: `"s3"`
- `framework`: `"hot"` or `"cold"` (storage class)
- `bucket_type`: `"private"` or `"public"` (default: `"private"`)
- `env_vars`: list of env var names the app needs (e.g. `S3_BUCKET`, `S3_ACCESS_KEY`)

## promto.yaml format

**IMPORTANT: promto.yaml is ALWAYS located at `/home/user/promto.yaml`.** This is the single infra config for the entire project — it describes ALL services: frontend, backend, databases, S3, etc.

```yaml
project_dir: my-project    # ← folder with code (relative to /home/user/)
services:
  service-name:
    friendly_name: "My App Frontend"
    description: "Short description"
    purpose: "Role of this service"
    order: 1
    type: frontend
    framework: react
    preset_id: 2785
    build_command: "npm install && npm run build"
    publish_dir: dist
    root_dir: ""
    env_file: ".env"
    env_vars: []
```

**Top-level fields:**
- `project_dir`: **(REQUIRED for code services)** Path to the project directory relative to `/home/user/`. This is the directory that gets pushed to the git repo for deployment. For example, if code lives at `/home/user/cat-shop/`, set `project_dir: cat-shop`. Database and S3 services don't need this (they're managed via API, no code push).

**Service fields:**
- `friendly_name`: Human-readable name shown in the Publish tab UI (e.g. "Сайт VibeCoding", "API сервер", "База данных PostgreSQL"). If omitted, the service key is used.
- `order`: Publish order (1 = first, 2 = second, etc.). Databases and S3 should have the lowest order so they are created before backends that depend on them. If omitted, defaults to: database=1, s3=1, backend=2, frontend=3.
- `type`: `"frontend"`, `"backend"`, `"database"`, or `"s3"`
- `framework`: For frontend — detected framework (`react`, `preact`, `vue`, `angular`, `svelte`, `nuxt`, `next.js`, `ember`, `another`). For backend — always `"docker"`. For database — engine name (`"postgresql"`, `"mysql"`, `"mongodb"`, `"redis"`, `"clickhouse"`, `"kafka"`, `"rabbitmq"`, `"opensearch"`). For S3 — storage class (`"hot"` or `"cold"`).
- `preset_id`: (optional) Timeweb preset ID that defines the resource tier (CPU, RAM, disk). If omitted, the cheapest available preset is used. See the preset tables above for available IDs per service type.
- `build_command`: Command to build the service
- `start_command`: (backend only) Command to start the server
- `publish_dir`: (frontend only) Output directory (e.g. `"dist"`, `"build"`)
- `bucket_type`: (S3 only) `"private"` or `"public"` (default: `"private"`)
- `root_dir`: Subdirectory if the service is not at project root (empty string if at root)
- `env_file`: Path to the environment variables file, relative to the project directory defined by `project_dir` (e.g. `".env"`, `".env.production"`, `"api/.env"`, `"backend/.env.local"`). **IMPORTANT**: Always set this to the actual env file used by the service. Different services may use different env files. Defaults to `".env"` if omitted.
- `env_vars`: List of required environment variable names

**Where to create `.env` files:**
The `.env` file path is always: `/home/user/{project_dir}/{env_file}`. For example:
- `project_dir: cat-shop`, `env_file: api/.env` → create at `/home/user/cat-shop/api/.env`
- `project_dir: cat-shop`, `env_file: .env` → create at `/home/user/cat-shop/.env`
- `project_dir: my-project`, `env_file: backend/.env.local` → create at `/home/user/my-project/backend/.env.local`

**NEVER create `.env` at `/home/user/.env`** — it must always be inside the `project_dir` folder. The platform reads env files from `{project_dir}/{env_file}` during deployment and syncs them to the hosting provider.

## Step-by-step publishing workflow

When the user asks to publish or deploy the project, follow these steps exactly:

### Step 1: Prepare the deployment configuration

1. Read this file (`/home/user/.agents/skills/инструкции_по_публикации.md`) to understand the `promto.yaml` format and requirements for the project's stack.
2. Create or update `/home/user/promto.yaml` with the correct service configuration.
   - **IMPORTANT:** promto.yaml is ALWAYS at `/home/user/promto.yaml` — never inside project subdirectories.
   - Set `project_dir` to point to the folder with code (e.g. `project_dir: my-project`).
   - Always make sure this file is up-to-date with the current project state.
3. For Docker-based services (backends AND SSR frontends), ensure a `Dockerfile` exists following these rules:
   - **Single-service project:** Dockerfile in project root is OK.
   - **Multi-service project:** Dockerfile MUST be inside each service's subdirectory (e.g. `api/Dockerfile`, `Go-back/Dockerfile`, `web/Dockerfile`). **NEVER put Dockerfiles in the project root** for multi-service projects — it causes conflicts. Set `root_dir` in promto.yaml to the service's subdirectory.
4. Commit all changes (including the `promto.yaml` file). The Publish tab reads directly from the sandbox filesystem — no git push is needed.

### Step 2: Guide the user to the Publish tab

Tell the user:
> Конфигурация готова! Перейдите во вкладку **Публикация**.

If the user was already on the Publish tab, tell them to refresh:
> Нажмите кнопку **⋯** (три точки) в правом верхнем углу вкладки Публикация и выберите **Обновить**, чтобы подтянуть новую конфигурацию.

### Step 3: Service cards appear

After refreshing, the Publish tab will parse `promto.yaml` and show a **service card** for each service defined in the file. The user should see their services listed with type, framework, and status.

If no service cards appear, the `promto.yaml` file is likely missing or malformed. Double-check the file location and format.

### Step 4: User clicks Publish

The user clicks the **Опубликовать** button on the service card (or **Обновить** if the service was previously published). The platform handles the actual build and deployment — do NOT try to deploy via API or CLI.

### Step 5: Debugging deployment errors

If the user reports a publish or build error:

1. Read `/home/user/История_Публикаций/<friendly_name>.log` — each service has its own deployment log file named by its `friendly_name` from `promto.yaml`. The file is overwritten on every deployment attempt with the latest logs. For example, a service with `friendly_name: "API сервер"` will have its log at `/home/user/История_Публикаций/API сервер.log`. If no friendly_name is set, the service key name is used (e.g. `api.log`).
2. Diagnose the error from the logs.
3. Fix the code, update `promto.yaml` if needed, and commit.
4. Tell the user to click **Обновить** in the Publish tab to re-deploy.

**Common issues:**
- Build fails → check `build_command`, missing dependencies, wrong `publish_dir`
- Backend won't start → check `Dockerfile`, missing `EXPOSE`, app not binding to `0.0.0.0`
- No services shown → `promto.yaml` is missing or has invalid JSON
- Stale config → user needs to refresh the Publish tab (⋯ → Обновить)

## Important rules

- Do NOT try to deploy via API, CLI, or any external service. The Publish tab handles everything.
- `promto.yaml` is ALWAYS at `/home/user/promto.yaml` — the single infra config for all services.
- **CRITICAL: Only ONE `promto.yaml` must exist — at `/home/user/promto.yaml`.** Before creating or updating `promto.yaml`, the agent MUST verify there are no other `promto.yaml` files anywhere in the filesystem (e.g. inside project subdirectories like `/home/user/my-project/promto.yaml`). Run `find /home/user -name "promto.yaml"` to check. If any duplicates are found, DELETE them immediately — only `/home/user/promto.yaml` is valid. Stray copies inside project directories can confuse the platform and cause deployment failures or stale config being picked up.
- **CRITICAL: `project_dir` field is REQUIRED** when you have frontend or backend services. It tells the platform which folder to push to the git repo. Without it, the entire `/home/user/` directory gets pushed (including system files), causing build failures.
- **Multi-service Dockerfile rule:** NEVER place Dockerfiles in the project root when there are multiple services. Each Docker-based service (backend or SSR frontend) gets its own Dockerfile inside its `root_dir` subdirectory.
- **Docker Compose forbidden directives:** `cap_add`, `cgroup`, `cgroup_parent`, `devices`, `privileged`, `device_cgroup_rules`, `volumes`, `ipc`, `pid`, `security_opt`, `userns_mode`, `sysctls`, and network mode `host` — all will cause deploy errors.
